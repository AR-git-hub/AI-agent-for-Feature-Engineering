## Мультиагентная система генерации признаков

### Общая идея

Система состоит из 5 агентов на базе GigaChat-2-Max и 13 инструментов (7 у оркестратора, 6 у аналитика). Задача — автоматически прочитать табличные данные, понять их структуру, придумать до 5 признаков для бинарной классификации, реализовать их и сохранить результат. Всё должно уложиться в 600 секунд.

Ключевой принцип архитектуры — **разделение когнитивных задач**. Каждый агент делает одну вещь и делает её хорошо. Аналитик не пишет код, генератор не оценивает качество, кодер не придумывает идеи, критик не лезет в данные. Оркестратор не делает ничего из перечисленного — он только управляет последовательностью.

---

### Агенты

#### 1. Оркестратор

Центральный агент, менеджер пайплайна. Он никогда не видит сырые данные, не читает csv, не пишет код. Работает исключительно с абстракциями: "отчёт аналитика готов", "генератор предложил 5 признаков", "критик оценил набор на 0.78".

У оркестратора 7 инструментов: `analyze`, `generate`, `merge`, `build_features`, `compute_stats`, `evaluate`, `save`. Каждый вызов инструмента — это обращение к суб-агенту (кроме `save`, который является чистой Python-функцией).

Оркестратор принимает все стратегические решения: запускать ли второй раунд генерации, есть ли время на уточняющий вопрос от критика к аналитику, какой набор признаков выбрать финальным. Он следит за бюджетом времени и может обрезать пайплайн, если осталось мало секунд.

Его системный промпт содержит общую инструкцию по пайплайну: порядок вызовов, критерии остановки, правила обработки ошибок. Промпт короткий и не содержит деталей о данных или признаках — всё это приходит от суб-агентов.

#### 2. Аналитик

Агент-исследователь данных. Единственный агент, который напрямую работает с файловой системой `data/`. У него 6 собственных инструментов, и он вызывает их итеративно — как живой дата-сайентист, который сначала смотрит общую картину, потом углубляется в детали.

Аналитик получает от оркестратора одну из двух задач: "исследуй данные с нуля" (в начале пайплайна) или "ответь на конкретный вопрос" (когда критик запросил уточнение). В первом случае аналитик проходит полный цикл исследования. Во втором — делает точечный запрос через свои инструменты и возвращает короткий ответ.

Результат работы аналитика — структурированный JSON-отчёт, который содержит: список таблиц и их описание, ключи и связи между таблицами, имя колонки-таргета и колонки-идентификатора, типы полей (числовые, категориальные, даты), статистику пропусков и выбросов, рекомендации по джойнам, потенциальные проблемы (дубликаты, many-to-many, data leakage).

Этот отчёт — основной документ для всех остальных агентов. Генератор опирается на него при придумывании признаков, кодер — при написании кода.

**6 инструментов аналитика:**

`read_readme` — читает файл `data/readme.txt` и возвращает его текст. Это первый инструмент, который вызывает аналитик. Из readme он узнаёт, что за задача, какие таблицы есть, как они связаны, что является таргетом. Без этого описания аналитик работал бы вслепую.

`list_tables` — сканирует папку `data/` и возвращает список всех csv-файлов с базовой метаинформацией: количество строк и столбцов, размер файла, разделитель. Аналитик вызывает его после `read_readme`, чтобы сопоставить описание из readme с реальными файлами. Может обнаружить файлы, не упомянутые в описании, или наоборот — отсутствующие файлы.

`table_info` — принимает имя конкретной таблицы и возвращает сжатую статистику: dtypes каждой колонки, количество пропусков, количество уникальных значений, describe() для числовых полей. Не возвращает сами данные — только метаинформацию. Аналитик вызывает этот инструмент для каждой таблицы, которую считает важной. Может вызвать повторно после мержа, чтобы проверить результат.

`peek_rows` — показывает N строк из таблицы. Принимает параметры: имя таблицы, количество строк, режим (head, tail или sample). Аналитик сам решает, сколько строк ему нужно и откуда. Например, 3 строки из head для понимания формата, или 20 строк sample для обнаружения паттернов. Может вызывать многократно для разных таблиц или разных режимов.

`request_merge` — формулирует запрос на мерж двух таблиц. Аналитик указывает: какие таблицы мержить, по каким ключам, какой тип join (left, inner, etc.). Инструмент не выполняет мерж сам — он передаёт запрос оркестратору, который вызывает кодера через инструмент `merge`. Результат мержа (shape, статистика потерь строк, процент null в новых колонках) возвращается аналитику, и он может продолжить исследование на объединённых данных — например, вызвать `table_info` или `peek_rows` для результата.

`report` — финальный инструмент аналитика. Формирует структурированный JSON-отчёт и передаёт его оркестратору. Это явный сигнал "я закончил исследование". Без этого инструмента аналитик мог бы бесконечно вызывать `table_info` и `peek_rows`, сжигая время. `report` — это точка выхода.

#### 3. Генератор

Креативный агент. Его единственная задача — придумывать идеи признаков. Он не пишет код, не запускает ничего, не оценивает качество. Только генерирует описания на естественном языке.

На вход получает отчёт аналитика (всегда) и фидбек критика (во втором раунде). На выходе — список из 5 описаний признаков. Каждое описание содержит: имя признака, словесное описание логики вычисления, какие таблицы и поля нужны, гипотезу о полезности для таргета.

Почему генератор отделён от кодера: это две разные когнитивные задачи. Придумать "посчитать отношение суммы транзакций за 30 дней к среднему за год" — это креативная работа, требующая понимания домена и таргета. Написать `df.groupby('client_id').apply(lambda x: x[x['date'] > cutoff]['amount'].sum() / x['amount'].mean())` — это инженерная работа, требующая знания pandas и обработки edge cases. Когда LLM пытается делать оба в одном промпте, качество обоих падает.

Системный промпт генератора содержит инструкции по типам признаков, которые обычно хорошо работают с бустингом: агрегации по группам, отношения между числовыми полями, временные окна, частотное кодирование категорий, count-encoding, target-статистики по категориям. Промпт также предупреждает о типичных ошибках: data leakage через будущие данные, бесполезные признаки с одним уникальным значением, признаки с 90% пропусков.

Во втором раунде генератор видит, какие признаки из первого раунда оказались слабыми и почему (по оценке критика). Это позволяет ему скорректировать стратегию: если признаки на основе транзакций не сработали из-за пропусков, попробовать признаки на основе других таблиц.

#### 4. Кодер

Инженерный агент. Получает задачу на естественном языке и реализует её в Python-коде. Единственный агент с внутренним циклом отладки: написал код → запустил → получил ошибку → исправил → запустил снова (до 3 попыток). Оркестратор не участвует в этом цикле — он ждёт готового результата.

Кодер вызывается тремя разными инструментами оркестратора, и его поведение зависит от того, каким инструментом он вызван.

Через `merge` — получает инструкцию от аналитика: какие таблицы мержить, по каким ключам, какой тип join. Пишет `pd.merge(...)`, запускает, возвращает shape результата и диагностику: сколько строк потерялось, сколько дублей появилось, процент null в новых колонках. Результат мержа сохраняется на диск, чтобы аналитик мог исследовать его дальше.

Через `build_features` — получает описания признаков от генератора и отчёт аналитика со схемой данных. Пишет Python-код, который вычисляет каждый признак для train и test одинаковой логикой. Это самая сложная задача кодера: нужно правильно сделать джойны, обработать пропуски, убедиться, что логика одинаковая для обеих выборок, избежать data leakage. Возвращает DataFrame с признаками и базовые статистики (mean, std, null%).

Через `compute_stats` — получает DataFrame с признаками и таргетом. Считает метрики, которые нужны критику: корреляцию каждого признака с таргетом (Pearson и Spearman), mutual information, процент пропусков, количество уникальных значений, VIF (variance inflation factor) между признаками для оценки мультиколлинеарности. Возвращает таблицу метрик.

Кодер работает с файловой системой напрямую — читает csv из `data/`, сохраняет промежуточные результаты. Его контекст минимален: описание задачи + схема данных. Он не знает про общий пайплайн, про раунды, про критика. Только код и данные.

Внутренний цикл отладки: если `exec` вернул traceback, кодер получает его вместе со своим предыдущим кодом и пытается исправить. Максимум 3 попытки. Если после 3 попыток код не работает, кодер возвращает оркестратору сообщение об ошибке с описанием проблемы. Оркестратор может решить попросить генератора предложить альтернативный признак.

#### 5. Критик

Агент-оценщик. Заменяет CatBoost для внутренней проверки качества признаков — это экономит вычислительные ресурсы и время. Критик не обучает модель, а анализирует прокси-метрики и делает выводы на основе своего понимания того, что хорошо работает с бустингом.

На вход получает таблицу метрик от кодера (через `compute_stats`): корреляции, mutual information, пропуски, уникальность, VIF. На выходе — ранжирование признаков от лучшего к худшему, оценка каждого признака с обоснованием, общая оценка набора, и фидбек для генератора (если будет второй раунд).

Критик оценивает по нескольким критериям. Предиктивная сила — насколько признак коррелирует с таргетом и сколько информации несёт (mutual information). Качество данных — процент пропусков, количество уникальных значений, есть ли выбросы. Независимость — не дублирует ли признак другие признаки из набора (VIF). Подозрение на leakage — слишком высокая корреляция (> 0.95) может означать утечку таргета.

Критик может вернуть флаг `need_clarification` с конкретным вопросом к аналитику. Например: "признак X имеет корреляцию 0.92 с таргетом — это подозрительно, есть ли в данных временная колонка, по которой можно проверить leakage?" Оркестратор решает, есть ли время на вызов аналитика для ответа. Если да — аналитик отвечает, критик пересматривает оценку. Если нет — критик помечает признак как подозрительный и работает с тем, что есть.

Во втором раунде критик сравнивает два набора признаков (из первого и второго раунда) и выбирает лучший. Критерий — не один признак, а набор целиком: лучший набор — тот, где 5 признаков в совокупности дают максимальное покрытие информации при минимальной мультиколлинеарности.

### 7 инструментов оркестратора

`analyze` — вызывает аналитика. Передаёт либо команду "исследуй данные с нуля", либо конкретный вопрос от критика. Аналитик внутри себя итеративно вызывает свои 6 инструментов и возвращает JSON-отчёт.

`generate` — вызывает генератора. Передаёт отчёт аналитика и (во втором раунде) фидбек критика. Возвращает 5 описаний признаков на естественном языке.

`merge` — вызывает кодера в режиме мержа таблиц. Передаёт инструкцию от аналитика: какие таблицы, какие ключи, какой join. Возвращает диагностику мержа.

`build_features` — вызывает кодера в режиме построения признаков. Передаёт описания от генератора и схему данных от аналитика. Возвращает DataFrame с готовыми признаками.

`compute_stats` — вызывает кодера в режиме подсчёта метрик. Передаёт DataFrame с признаками и таргетом. Возвращает таблицу статистик для критика.

`evaluate` — вызывает критика. Передаёт таблицу метрик от `compute_stats`. Возвращает ранжирование, оценки и фидбек.

`save` — единственный инструмент без LLM. Чистая Python-функция, которая берёт финальный набор признаков и записывает `train.csv` и `test.csv` в папку `output/`. Проверяет, что структура колонок совпадает, что ID и таргет на месте, что признаков не больше 5.

### Пайплайн работы

#### Раунд 1

Шаг 1. Оркестратор вызывает `analyze`. Аналитик читает readme, сканирует таблицы, исследует структуру, при необходимости запрашивает мерж через `request_merge` (оркестратор перенаправляет на `merge` → кодер). Аналитик возвращает отчёт.

Шаг 2. Оркестратор вызывает `generate`. Генератор получает отчёт аналитика и придумывает 5 описаний признаков.

Шаг 3. Оркестратор вызывает `build_features`. Кодер получает описания и схему, пишет код, запускает, дебажит, возвращает DataFrame с признаками.

Шаг 4. Оркестратор вызывает `compute_stats`. Кодер считает метрики по признакам.

Шаг 5. Оркестратор вызывает `evaluate`. Критик анализирует метрики, ранжирует признаки, даёт фидбек.

#### Раунд 2 (если есть время)

Оркестратор проверяет оставшееся время. Если осталось больше ~200 секунд, запускает второй раунд.

Шаг 6. Если критик запросил уточнение (`need_clarification`) — оркестратор вызывает `analyze` с конкретным вопросом. Аналитик отвечает. Критик пересматривает оценку.

Шаг 7. Оркестратор вызывает `generate` с фидбеком критика. Генератор предлагает улучшенные 5 признаков.

Шаг 8–9. `build_features` → `compute_stats` для нового набора.

Шаг 10. `evaluate` — критик сравнивает два набора и выбирает лучший.

#### Финал

Шаг 11. Оркестратор вызывает `save` с лучшим набором. Записываются `output/train.csv` и `output/test.csv`.

### Обработка ошибок

Ошибка кодера (код не работает после 3 попыток) — кодер возвращает описание проблемы оркестратору. Оркестратор может попросить генератора предложить альтернативный признак взамен сломавшегося и повторить `build_features`.

Ошибка аналитика (не смог прочитать таблицу, неожиданный формат) — аналитик возвращает отчёт с пометкой о проблемной таблице. Генератор учитывает это и не предлагает признаки на основе недоступных данных.

Таймаут — оркестратор следит за временем. Если остаётся менее 60 секунд, он прерывает текущий раунд и сразу вызывает `save` с лучшим доступным набором. Лучше сохранить 3 признака из первого раунда, чем не сохранить ничего.

Критик не может оценить (все метрики плохие) — критик возвращает оценку как есть, но помечает набор как "low confidence". Оркестратор всё равно сохраняет его — плохие признаки лучше, чем ноль признаков.

### Бюджет времени (примерная раскладка на 600 секунд)

Анализ данных (аналитик + мерж): ~100 секунд. Генерация идей (раунд 1): ~30 секунд. Построение признаков (кодер, включая дебаг): ~80 секунд. Подсчёт метрик: ~30 секунд. Оценка критиком: ~30 секунд. Итого раунд 1: ~270 секунд. Раунд 2 (если запускается): ~170 секунд. Запас на ошибки и повторы: ~160 секунд.
